# NanoFormulationDesigner

**A local, materials-aware decision-support workbench for early nanocarrier formulation design.**

NanoFormulationDesigner converts a proposed API–excipient composition into transparent, database-backed formulation descriptors, ranked CQA tendencies, batch tables, and a grounded laboratory screening plan. It is built for preformulation scientists who want a reproducible first-pass design space before committing materials, instrument time, and experimental runs.

> **Scientific boundary.** The platform is not a validated predictor, not a medical device, and not a substitute for experimental formulation development. Outputs are descriptor-driven screening estimates whose confidence depends on the underlying material-property coverage.

---

## Why this project exists

Early nanomedicine formulation work often begins with scattered excipient constants, informal heuristics, and hard-to-audit spreadsheet logic. This project makes that first screening step explicit: constants live in a curated local database, calculations live in deterministic kernels, and each recommendation exposes its assumptions, missing values, and risk drivers.

The result is not a “best formulation generator.” It is a defensible formulation triage system: useful for deciding what to test first, what to avoid, and which missing constants must be curated before a design can be trusted.

## Core capabilities

- Deterministic kernels for batch mass, mixed HLB, Tanford chain geometry, critical packing parameter (CPP), Hansen distance / RED, Hildebrand parameter, Flory-Huggins χ, and ionization fraction.
- A local material-property database with relational CSV tables, derived wide constants, source provenance, confidence scores, and a prioritized missing-value queue.
- A formulation designer that returns CQA tendencies, an executive decision, design maturity level, descriptor table, batch-mass table, explanations, improvement suggestions, sanity checks, and a laboratory screening plan.
- Solvent and carrier recommenders with route/process constraints.
- Guided starting-point generator, multi-design comparator, Streamlit interface, command-line interface, and Markdown report export.
- Optional LLM-assisted report polishing that is disabled by default and restricted to grounded evidence bundles generated by the deterministic layer.

## What it does not claim

- It does not predict numeric encapsulation efficiency, particle size, PDI, zeta potential, in-vivo exposure, biodistribution, toxicity, or stability shelf-life.
- It does not replace DoE, preformulation experiments, analytical validation, sterility/endotoxin control, or regulatory assessment.
- It does not depend on live external datasets at runtime.
- It does not fabricate constants when database coverage is incomplete; missing values are surfaced and logged.

## Supported formulation families

Niosomes, proniosomes, liposomes, transfersomes, ethosomes, bilosomes, lipid nanoparticles, solid lipid nanoparticles, nanostructured lipid carriers, nanoemulsions, polymeric nanoparticles, lipid–polymer hybrids, and dry-powder/carrier systems across oral, topical, transdermal, ocular, pulmonary, nasal, and parenteral routes.

## Installation

```bash
python -m venv .venv
# Windows: .venv\Scripts\activate
# macOS/Linux: source .venv/bin/activate

pip install -e .
# For development:
pip install -e ".[dev]"
```

The package ships with a runtime copy of the CSV database under `src/nanoform/data/`. When running from the repository root, the tool uses the maintainer-facing `data/` directory. To use another database location, set:

```bash
export NANOFORM_DATA_DIR=/path/to/data
```

## Run the app

```bash
streamlit run app.py
```

The app contains ten tabs: Dashboard, Guided Wizard, Formulation Builder, Solvent Recommender, Carrier Recommender, Compare Designs, Material Explorer, Custom Material, Reports, and AI Orchestration.

## Use the CLI

```bash
python -m nanoform.cli search --query "Span 60"

python -m nanoform.cli design --drug Dexamethasone --family niosome --route topical \
  --goal stability --solvent Ethanol \
  --components "Span 60:surfactant:47.5|Cholesterol:sterol:47.5|Dicetyl phosphate:charge_inducer:5"

python -m nanoform.cli recommend-solvent --drug Curcumin --route oral --blends
python -m nanoform.cli recommend-carrier --route pulmonary --family dry_powder_carrier
python -m nanoform.cli wizard-candidates --drug Dexamethasone --route topical --family niosome
python -m nanoform.cli cqa-table --drug Curcumin --family solid_lipid_nanoparticle \
  --components "Glyceryl behenate:solid_lipid:80|Tween 80:surfactant:20"
python -m nanoform.cli design-report --drug Doxorubicin --family liposome \
  --components "HSPC:phospholipid:56|Cholesterol:sterol:39|DSPE-PEG2000:peg_lipid:5" \
  --out report.md
```

## Example Python use

```python
from nanoform.designer import DesignInput, design
from nanoform.reporting import build_markdown_report

inp = DesignInput(
    family="liposome",
    route="parenteral",
    design_goal="parenteral_cautious",
    drug="Doxorubicin",
    drug_mol_percent=5,
    solvent="Ethanol",
    carrier="Sucrose",
    components=[
        ("HSPC", "phospholipid", 56),
        ("Cholesterol", "sterol", 39),
        ("DSPE-PEG2000", "peg_lipid", 5),
    ],
)

result = design(inp)
print(result.executive_decision, result.nanoform_score)
print(build_markdown_report(result))
```

## Database model

The database is material-property based, not a library of copied formulations. It stores the constants needed to calculate a candidate composition: molecular weight, HLB, charge, chain length, transition temperature, Hansen parameters, headgroup area, molar volume, ICH solvent class, carrier flags, and related descriptors. Formulation/outcome tables are reserved for user-created history and future internal experimental calibration.

Current generated database snapshot:

- 321 materials
- 3,449 property rows
- 25 source records
- 313 prioritized unresolved missing-value rows

Curation rule: **do not fabricate precise values.** Stable constants are marked as literature/database-backed where available. Grade-dependent, model-derived, or uncertain values are explicitly tagged with reduced confidence and surfaced to the user.

## Design maturity

The current build is capped at **DML-2: descriptor-supported rational design** because no internal experimental outcome dataset is bundled. DML-3 would require retrospective calibration against internal experimental outcomes. DML-4 would require prospective validation.

## Optional LLM layer

The optional LLM layer is deliberately narrow: it can polish or summarize evidence already produced by the deterministic system, but it cannot introduce constants or override the calculation layer. With no model client configured, the platform returns deterministic summaries and reports.

See [`docs/AI_ORCHESTRATION.md`](docs/AI_ORCHESTRATION.md).

## Rebuild and validate the database
